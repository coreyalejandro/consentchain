## Phase 3: Develop and Run the Reasoning Architecture Experiment

**Objective:** Implement your new reasoning architecture step-by-step, conduct experiments, and collect results, all while managing attention and avoiding errors.

* **Research Briefly First:** Spend a focused half-day on a **literature scan** to ground your approach. Give me a few examples of papers or known techniques.”_ This might surface terms like “chain-of-thought prompting” or “self-consistency in reasoning”. Note any references to cite later or ideas to incorporate (but don’t get bogged down in reading dozens of papers – limit this to key insights). This step boosts credibility: showing awareness of related work in your paper/application.

* **Design the Architecture (Theory):** On paper or a whiteboard (or a fresh Claude chat), outline how your reasoning system will work. For example: _“Step 1: Model A generates a draft reasoning chain. Step 2: Model B (or same model) evaluates or verifies each step. Step 3: If a step is flawed, the model revises it (maybe using a feedback loop). Step 4: Final answer is produced with an explanation.”_ Define the roles clearly. Because spatial reasoning for flowcharts might be tricky, do this in a modality that suits you: you can write it as a numbered list (as above), which is perfectly fine. If you do want a visual, consider using **MermaidJS** or a simple online flowchart tool – but only if you find it helpful, not frustrating. The goal is to have a **clear mental model** of the process.

* **Select a Test Problem Domain:** Choose a task to test your architecture on. Ideally, something requiring reasoning: e.g. math word problems, logical puzzles, or a safety scenario (perhaps a simplified alignment problem where reasoning is needed). Keep it simple enough to implement in the time frame. For example, you might use a set of common reasoning puzzles or a subset of a benchmark like GSM8K (math problems) to see if your architecture improves accuracy. Alternatively, design a few custom questions that you think will highlight the benefits of your approach (perhaps comparing answers with vs. without the explicit reasoning steps). List \~5–10 test problems for a start.

* **Implement the Prototype:** Start coding the experiment. This could be a Python script that uses an API (OpenAI or Anthropic) to run two conditions: **baseline vs. your architecture**. For baseline, you might just ask the model the question directly. For your architecture, you’ll prompt it step-by-step (or orchestrate multiple prompts). For instance, you can programmatically do: call model for step 1 (get reasoning), feed that into step 2 prompt, etc. Given time constraints, aim for a **functional prototype** rather than a perfect system. Comment your code for clarity (helps if Claude or anyone reviews it). If coding feels overwhelming, you can also simulate this manually for a few examples (having Claude step through reasoning in chat for one method and comparing to a one-shot answer) – but coding will give you systematic results.

* **Test Incrementally:** As you build, test each part immediately. For example, after coding Step 1 (generate reasoning chain), run it on a sample question and inspect the output. Does it make sense? If not, adjust your prompt or approach. Use Claude as a coding assistant if you run into bugs: _“Here’s my code and error, what did I do wrong?”_. Keep your interactions focused – one issue at a time – to avoid confusion.

* **Run the Experiment & Collect Data:** Once the prototype works end-to-end, run it on your set of test problems. Collect the outputs for both baseline and your method. Save these results in a structured way: for example, a CSV or JSON with columns like `question`, `baseline_answer`, `newMethod_answer`, `baseline_correct?` etc. Also save the reasoning traces if possible (these might be qualitative but you can later pick illustrative examples). If any run fails or looks odd, make a quick note but don’t derail – you can circle back if time permits.

* **Manage Focus During Runs:** Running experiments can be repetitive, which is where ADHD might cause distraction. Use **timers or the Pomodoro technique** here: e.g. set a 25-minute timer to run and monitor experiments, then a 5-minute break. During runs, resist the urge to context-switch to something else; use that break time for any digressions (like checking an interesting side output or tidying code). Conversely, if you hit a frustrating bug (triggering OCD perfectionism), step away for 5 minutes – clear your head or quickly describe the issue to Claude for help. This prevents getting stuck in a loop.

* **Validate and Preempt Errors:** Double-check critical parts of your experiment for accuracy to avoid false results. For example, ensure that when you evaluate whether an answer is correct, you’re doing it fairly (if it’s a math problem, maybe code an automatic checker or at least manually verify a few). If using an AI to judge correctness, be cautious – an AI might err. It’s okay if not all results are perfect; you can mention limitations later. The key is to not publish incorrect claims. Preempt this by spot-checking.

* **Document Everything as You Go:** Keep a simple log of what you did each day (date, tasks done, any key findings). For instance: _“Dec 30: Implemented reasoning loop; initial test on puzzle X showed the AI’s chain-of-thought included a factual error at step 3 – noted for analysis.”_ This diary will be gold when writing your paper and reflecting in your personal statement. It also helps you verify that each part of the plan was completed.

* **Stay Grounded in Safety Motivation:** As you develop, periodically remind yourself (even out loud or in writing) _why_ this project matters to you and to AI safety. E.g., _“If this works, it could be a step toward AI systems that explain themselves, which is critical for safety and trust.”_ This can be motivating and also will feed good material into your application essays. You might uncover personal insights here (“I care about this because…”) – write those down; they can strengthen your narrative later.

**Deliverable for Phase 3:** A working experimental setup and a collection of raw results. By the end of this phase, you should have: code (or a reproducible procedure), output data (saved answers or logs), and initial observations on how the new reasoning architecture performed. Even if the results are mixed or the prototype is rudimentary, you have the **core content for your research paper** and application (you can say you conducted an experiment and discuss findings).
